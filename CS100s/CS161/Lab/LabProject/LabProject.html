Project: Creating a Typing Game



Your final lab project will be to create a game that can be used to improve a user's typing skills. The basic idea of the game is as follows. Random words "float"  HORIZONTALLY across the screen. The user types the words into a JTextField and presses enter.

  • If the typist types the word correctly, points are awarded to the typist.
  • If the word typed does not match, or if the word floats off the screen prior to the typist typing the word, the typist receives a strike.

Once the typist receives three strikes, the game is over. We wish to have multiple degrees of difficulty for the game based on word length and the speed at which words appear and disappear off of the screen.

 

**************************************************************************************************

Part I Creating a random word generator (In lab)



In order to float words across a screen, we need to have a source of words of varying length. One way to do this is to take a large document and then use the words in that file as a source of words. The vbush.txt text file is available in your lab directory. (P.S.  The text file provided is an article written by Vannebar Bush in 1945.  In the article, he describes many of the concepts behind the World Wide Web when there were few computers in the world!).  From the command line we can use the wc command on file vbush.txt to find out how many words are in the file (i.e. wc vbush.txt).  The wc command prints the line, word, and byte counts for the specified file. Report the number of words in the file to your TA or lab instructor.

This file contains plenty of words, but is not quite ready for use. Since our game is intended to send individual words on the screen one at a time we will want to remove all punctuations such as commas (,) , periods (.) , colons (:), semicolons (;), apostrophes(‘), dashes(-), and exclamation (!) and question marks (?) as well other characters (such as /, @, (, ) and ~) that show up in the file. In addition, we wish to sort the words based on word size.

 

Step one in your game development is to create a WordSorter java class.  This class should have a public static void main(String [] args) method in order to run it from the command line with an input filename as an argument (for which you’ll provide vbush.txt).  It should generate three new files that will contain small (3-5 letters), medium (6-8 letters) and large (>8 letters) (ignore words of length 2 or less)  words where the words in the input file have been cleaned (i.e. filtered to remove the punctuation and other characters). You will be reading words from the input file one at a time, cleaning them, and writing them to one of the three files. Class String has method replaceAll(String match, String replacement) that can be used for cleaning purposes; e.g., to remove “@” from a String x, you can write x = x.replaceAll(“@”, “”). This replaces all matches of “@”with the empty string which effectively removes the matches. You can also include multiple characters between square brackets to remove all of them simultaneously; e.g., to remove /, @, % and $ all at once, you can use x = x.replaceAll(“[/@%$]”, “”)).

Next, create a RandomWord class that takes a filename as a parameter in its constructor and has a next() method that will return a word from the specified file at random. Do this by reading the words in the specified file into an ArrayList<String> and then generating a random location number using RandomIntGenerator class to find the next word to return. Class BannerMover uses a RandomIntGenerator to randomly access random words from the ArrayList<String> defined in its constructor; you might want to follow the steps outline there.

Write a short demo program that uses the RandomWord class and show your Lab Instructor or TA that you have successfully completed this part of the program.

Part II Creating an Appropriate GUI

The second step is to design the GUI that you intend to use. Remember that you need to have a way for the user to select the difficulty level, a canvas for the words to float across the screen, a place for the user to type, a place to show the score and the number of strikes (or number of strikes left). Also include a Start and an Exit button. You should first sketch your design on paper and show it to your Lab Instructor or TA and then develop the code that provides all the GUI components needed. The functionality behind the GUI components can be developed later.

Part III Sending Words Across the Screen

Some of the work for this part is done in that you have access to  the Banner and BannerMover classes.  You can use them and modify them as you like. Remember that you need to send multiple words across the screen (travel horizontally from left to right) without user action. The banner containing the word should be removed from canvas after the user types in his/her word and hits enter. You may want to look carefully at the RainCloud.java class from Chapter 9 where we designed a new ActiveObject class to create ActiveObjects of type FallingDroplets with appropriate pausing in between. A variation of this might help you send words across the screen. For now, send the words across the screen one at a time. This makes it easier to determine if a word typed matches the word on the screen.

 

Currently, class BannerMover uses a RandomIntGenerator to randomly access random words from the ArrayList<String> defined in its constructor. You need to change class BannerMover so that its constructor will take a word string and a speed. Have your created ActiveObject class pass to BannerMover speeds and random words based on the level of difficulty selected by the user (for easy level, use slow speed and the small words file; for medium level, use medium speed and the medium words file; and for hard level, use fast speed and the long words file).  You can use the RandomWord class created in Part I  to generate the words.

Part IV Putting It Together

You now need to tie everything together where actions are tied to the GUI. You will also need to determine whether or not the word typed by the user matches the word on a banner. Whenever this occurs, the word banner should be removed from the screen and a new word banner should be generated. If a word is mistyped or if a word floats off the screen and disappears, you should mark it against the user; otherwise, the user gets credit for typing the word correctly. Again, looking at the code in Ch 9 will give you some ideas on how you can remove an object before it reaches the right edge of the screen. The game should end when the user gets three strikes.
