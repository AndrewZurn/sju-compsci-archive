Part I: Using gedit

In this lab we are going to switch from DrJava to  gedit to edit our programs and the command line to do everything else. To start with, I have given you two sample files: SortInputs.java and StartSortInputs.java. On the command line, use cd to change into CS161/Lab16.  Open SortInputs.java by typing gedit SortInputs.java &  (The & symbol is necessary to allow you to use your shell terminal window while gedit is open). If you forgot the &, however, you can achieve the same result by going to the shell, holding down the control key while you hit z (<ctrl-z>) and, then, typing bg at the command line followed by a return. The & and the <ctrl-z> bg have the same effect - they make the program (gedit in this case) run as a concurrent process independent of the shell.).  Use the same shell window to open the file StartSortInputs.java (giving you both files at the same time).   Notice that the StartSortInputs.java class simply contains the main method which creates an object of type SortInputs and, then, runs its run() method.  StartSortInputs.java is used simply to lauch an instance of the SortInputs class.

When we write a program for use from the command line, we need a "launching" class that has a main method in it. In general, the "launching" class looks as follows:

public class RunSomeOtherClass {
      public static void main(String [] args) {
      SomeOtherClass someObject=new SomeOtherClass(....);
       someObject.run(); //or, perhaps, some other methods in the   class
                                    //SomeOtherClass might also be called.
      }
}
We will explore the main method further in Part VI of this lab, but for now, note that it will help us "run" a program from the command line.

Part II: Compiling and running from the command line

On the command line, use cd to change into CS161/Lab16  (if you’re not there already).  Type javac *.java to compile both of those java files (using * instead of a file name compiles all files that end in .java). This is equivalent to clicking on the compile button in DrJava for all of the files in the directory.  Type java StartSortInputs to run the program: type in a series of eight to ten numbers (you can put them all on the same line with spaces in between and then hit return or hit return after each entry.)  Then, hold down the control key and hit d. (This key stroke combination - written <ctrl-d> - generates a keyboard end of file and, therefore, stops scanner input from the keyboard.) You should now see your numbers typed out, one per line, in sorted order in the Terminal Window.

Part III: Filters and redirection

The program you created in part II is called a filter because it reads from standard input (which defaults to the keyboard) and writes to standard output (which defaults to the terminal where you launched the program).  The name filter is used because it "filters the data into a new form." The neat thing about filters in unix is that filters can be faked into reading from places other than the keyboard and writing to places other than the screen in a very simple way. To start, go back into gedit and create a new file called someNumbers (i.e. type gedit someNumbers & while in the same Lab16 folder). In file someNumbers type in a set of numbers not in sorted order and, then, save the file into your Lab16 directory. Then type java StartSortInputs < someNumbers on the command line and you should see as output the numbers from someNumbers in sorted order. The < sign is the standard input redirection symbol and it says to the java program that the input is going to come from the file someNumbers instead of the keyboard.

Now type java StartSortInputs > someSortedNumbers and type in a series of numbers not in sorted order followed by the <ctrl-d>. You won't see any output at all this time. Instead, the > sign has redirected the output into the file someSortedNumbers (which it created on the fly). Type ls to see that the file someSortedNumbers has been created.  Type cat someSortedNumbers to display the contents of the file and see your input printed one per line in sorted order. 

You can actually create data files in unix more easily than this. Type cat > evenMoreNumbers at the command line. Then type a series of numbers followed by the same old return followed by <ctrl-d>. This will create a file evenMoreNumbers because the cat command is a filter - i.e., if you don't tell it anything else, it uses standard input and output. Since we redirected standard output to evenMoreNumbers the numbers you typed in got copied into evenMoreNumbers. Type cat evenMoreNumbers to see the result.

Type java StartSortInputs < evenMoreNumbers > evenMoreSortedNumbers to sort the numbers in evenMoreNumbers and put them into the file evenMoreSortedNumbers. Type cat evenMoreSortedNumbers to verify the result.

There is one more "redirection" available. If, instead, of using > for redirecting output, you use >> then the output will be appended onto the end of an existing file (instead of rewriting the file from scratch). To try this, type java StartSortInputs < evenMoreNumbers > evenMoreSortedNumbers to see that you get an error trying to use ordinary redirected output to an existing file. Now type java StartSortInputs < evenMoreNumbers >> evenMoreSortedNumbers and then type cat evenMoreSortedNumbers to see that evenMoreSortedNumbers now contains the output from the first experiment followed by the output from the second experiment.

Part V: Pipes

Pipes are another redirection device that can be used to pipe the standard output of one program to the standard input of another. Its general form is command1 | command2 where the | is the key above the Enter on your keyboard. The way this works is that command1 and command2 run concurrently and as command1 outputs to standard output that output becomes input for command2. When command1 exits, that generates an end of file for command2 so it knows there is no more input. You can try this out on your own two filters - StartSortInputs and StartSumInputs - by typing java StartSortInputs | java StartSumInputs at the keyboard and, then, enter the series of numbers you wish to use. After you do the return followed by <ctrl-d> you should see the sum of the numbers printed out. Add file redirection to this piped sequence to see that you can still use redirected standard input for data (into StartSortInputs) and redirected standard output for the sum from StartSumInputs. Since they are standard input, you can use one of your input files into StartSortInputs and output from StartSumInputs to some other file.

Part VI: Command Line Parameters

As mentioned earlier in the lab, when we write a program for use from the command line, we need a "launching" class that has a main method in it. In general, the "launching" class looks as follows:

public class RunSomeOtherClass {
      public static void main(String [] args) {
      SomeOtherClass someObject=new SomeOtherClass(....);
       someObject.run(); //or, perhaps, some other methods in the   class
                                    //SomeOtherClass might also be called.
      }
}
Here, we are going to discuss the use of the String [] args inside the parentheses - the parameters for the main method. It turns out that the parameter args is what it says it is! That is, it is an array of Strings. More importantly, it is an array of Strings, each of which is something you typed on the command line after launching the program. For example, you would normally expect to type:

> java RunSomeOtherClass

to cause this program to run. If you did, the args array would be an array with 0 elements in it. On the other hand, if you typed

> java RunSomeOtherClass firstthing secondthing

the args array would be of length 2 and args[0] would be the string "firstthing" and args[1] would be the string "secondthing". What this means is that you can design your programs so that when you launch them from the command line you can specify things you want them to do.

We will do an example.  Suppose we wanted to specify a file for our program to read data from.  Then we could specify the file name at the command line. The user would, then, be able to run the program from the command line and specify the file when starting the program.
I have supplied you with a partially completed class called SumFile.java which has a constructor that expects a file name as a parameter and which then creates a Scanner designed to read from that file. I have also supplied two data files firstNumbers and secondNumbers. What you need to do is complete the run() method so that it reads and adds up the numbers read by the Scanner sc and, then, outputs the sum to standard output. Then complete the program RunSumFile so that it creates an object of type SumFile and passes args[0] as the parameter to the constructor and, then, calls the run() method on the class.

Run the program twice:

>java RunSumFile firstNumbers

and

>java RunSumFile secondNumbers

to verify that it is working properly. The output from the two runs should say something like

The sum of the numbers in file firstNumbers is 30.
The sum of the numbers in file secondNumbers is 45.


Part VII: JApplets


We have seen two ways to run programs: from inside DrJava and from the command line. There is a third way.  Java was designed to be used by Web browsers so that you could add "active" programs to the Web. In order to do this, you must do two things. First, you must create your program by extending something called JApplet and include a public void init() method to initialize the applet.. Second, you must launch it with an html file - a special file understood by Web browsers. This html file typically contains at least one line like the following:

<applet code="AJApplet.class" width="300" height="100"> </applet>

The code portion of this applet tag must point to the compiled version (i.e. .class) of your JApplet Java class. The width and height specify the size of your Java program on the Web page. Finally you must put it somewhere where the Web server can access it. On our Linux machines this last place is a directory in your home directory called www-docs. If you have such a directory, you are all set. If not, you can create one by simply typing cd ~ (i.e. go to your home folder) at the command line and, then, typing mkdir www-docs to create it. I have included a sample JApplet called AJApplet.java and an html file called AJApplet.html to launch it. At the command line type javac AJApplet.java and, then, copy all three files - AJApplet.java, AJApplet.html, and AJApplet.class - to the directory www-docs. Make sure that all these things are accessible by the browser by typing:

>chmod o+rx www-docs

and, then, changing into www-docs (i.e., cd www-docs) and typing:

>chmod o+rx *

to give necessary permissions to all files. Finally, cd out of your home folder:         

>cd ~/..

and type this on the terminal:

>chmod o+x your_user_id

(where your_user_id is your user id for Linux) This will enable the browser to access your www-docs.

Now, open firefox or mozilla and type into the white box at the top http://www.users.csbsju.edu/~ your_user_id/AJApplet.html (where your_user_id
is your userid for Linux) to see it run. Click on the buttons to see it work.
